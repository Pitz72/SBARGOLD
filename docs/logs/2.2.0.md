# 🪵 Log 22K.BULLIONE-2 - Security & Core Hardening

**Data**: 2026-03-17
**Versione**: 22K.BULLIONE-2 (Security Update)
**Precedente**: 22K.BULLIONE-1
**Autore**: Antigravity / Gemini

## 🚨 Sintesi
La versione 22K.BULLIONE-2 è un aggiornamento critico orientato alla sicurezza e all'integrità del sistema. Durante una revisione forense del codice sorgente dell'interprete (22K.BULLIONE-1) sono state individuate due vulnerabilità classificate come "Gravissime". Questo aggiornamento mitiga completamente tali falle senza alterare la sintassi e l'usabilità del linguaggio SBARGOLD.

## 🛡️ Risoluzione Criticità Gravissime

### 1. Prevenzione Arbitrary File Access (Directory Traversal)
- **Vulnerabilità**: I comandi `SBARGOLD>>` (Scrittura), `SBARGOLD<<` (Lettura) e `SBARGOLD|` (Importazione) accettavano percorsi file grezzi senza validazione. Ciò permetteva l'esecuzione di attacchi di Path Traversal (es. `../../../../etc/passwd`), esponendo il sistema host a fughe di dati sensibili o sovrascrittura di file di sistema.
- **Soluzione**: Implementato un **Virtual File System Sandbox**. È stato introdotto il metodo `_safe_path(user_path, node)` nella classe `Interpreter`. Questo metodo risolve il path in modo assoluto e garantisce, tramite un rigoroso string matching (`startswith`), che il file richiesto non esca dalla directory radice di esecuzione (`base_path`). I tentativi di evasione generano ora un `RuntimeError` protetto.

### 2. Mitigazione Resource Exhaustion (DoS da Ricorsione e Loop)
- **Vulnerabilità**: L'interprete non imponeva limiti alla quantità di istruzioni eseguibili o alla profondità dello stack di chiamata delle funzioni. Un attaccante o un programma mal scritto poteva lanciare chiamate ricorsive infinite o loop infiniti, causando esaurimento della memoria (OOM) o saturazione totale della CPU (Denial of Service).
- **Soluzione**: Implementazione di contatori hardware-level emulati:
  - **`MAX_CALL_DEPTH`**: Impostato a 1000. Gestito tramite un blocco `try/finally` in `_call_function` per prevenire Stack Overflow.
  - **`MAX_INSTRUCTIONS`**: Impostato a 5.000.000. Il ciclo `_execute` monitora ora un `instruction_count` globale. Questo funge da watchdog termico per prevenire infinite loop (sia tramite `SBARGOLD~` che tramite codice offuscato), garantendo la terminazione dell'interprete entro un range sicuro.

## 📈 Impatto Architetturale
L'interprete SBARGOLD è ora "Secure-by-default" e adatto ad essere eseguito in ambienti cloud automatizzati o servizi web (es. playground online) con un rischio marginale per l'infrastruttura ospitante. Le modifiche non introducono breaking changes retroattive, eccetto per script malevoli preesistenti.
